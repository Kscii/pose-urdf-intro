# 

# **base\_link 坐标系下 End Pose 计算与校验方案**



## **1\. 概念说明**

### **1\.1 坐标系与 Pose**

机器人数据中的位姿\(pose\)由两部分组成：

- position：位置，通常表示为 \[x, y, z\]。

- orientation：姿态，当前方案中统一使用四元数 xyzw, 只有在URDF和foxglove中使用rpy\(roll, pitch, yaw\)表示。

一个 pose 必须依赖某个坐标系才有意义。如果它是在 base\_link 坐标系下，含义是“该点相对机器人 base\_link\(通常是机器人本体中心的一个参考点\) 的位置”



### **1\.2 常见机器人坐标系**

本方案中主要涉及以下几类坐标系：

|坐标系|含义|
|---|---|
|world|位于机器人本体之外的世界坐标系，坐标系不会因为机器人本体的运动而改变|
|base\_link|机器人本体根坐标系，通常固定在机器人底盘或躯干根部。本方案中的所有end pose都会被归一化到base\_link坐标系下|
|link frame，例如 left\_arm\_link3|URDF 中某个 link 对应的坐标系。urdf中的joint的origin字段就是基于其parent\_link的坐标系下的pose变换|
|raw\_pose\_frame|部分厂商定义的 end pose 所基于的坐标系，不一定是base\_link, 这种情况需要归一化到使用base\_link|



### **1\.3 URDF说明**

URDF 是机器人结构描述文件，主要描述机器人各个部件之间的连接关系。这里主要关注三个概念：

- link：机器人上的一个刚体部件，例如 base\_link、left\_arm\_link6、left\_gripper\_link。

- joint：连接两个 link 的关节，定义 parent link \-\> child link 的关系。

- joint type：包含 fixed、revolute、continuous、prismatic。

```Plain Text
# urdf的例子

      upper="1.518"
      effort="0"
      velocity="0" />
  </joint>
```

因此一个 joint 在fk计算中需要使用的信息其实只有以下这些：

- parent link

- child link

- origin\_xyz

- origin\_rpy

- axis

- type



其中：

- origin\_xyz 表示 joint frame 相对 parent link 的固定平移。

- origin\_rpy 表示 joint frame 相对 parent link 的固定旋转。

- axis 表示关节运动轴，只对可运动关节有实际作用，实际所有的构型的所有joint都是沿着xyz中的一个轴旋转或者平移的，不会出现斜轴的情况。

    - revolute / continuous：绕该轴旋转。

    - prismatic：沿该轴平移。

    - fixed：不使用 axis。

### **1\.4 End Pose 的分类**

当前方案中会被计算并写入h5和报告的end有以下5类，这些类型也在配置文件中有体现：



|类型|来源|用途|
|---|---|---|
|raw\_action\_end|H5 中厂商提供的 action end pose|作为厂商 action 末端数据，归一化后用于对比，目前只有少量构型拥有|
|raw\_state\_end|H5 中厂商提供的 state end pose|作为厂商 state 末端数据，归一化后用于对比，目前只有少量构型拥有|
|verify\_action\_end|使用 action 关节数据通过 FK 计算得到|用于和 raw\_action\_end 做误差对比，所有有提供正确urdf以及完整action joint自由度话题的构型都可以计算出来|
|verify\_state\_end|使用 state 关节数据通过 FK 计算得到|用于和 raw\_state\_end 做误差对比，所有有提供正确urdf以及完整state joint自由度话题的构型都可以计算出来|
|tcp\_end|使用 state FK 加人工 TCP 偏置得到|表示真实物理抓取点或操作点，需要人工标定，有关标定方法需要后续进一步讨论|

其中：

raw\_\* = 厂商原始 end pose
verify\_\* = 基于 URDF 和关节数据计算出来的诊断 end pose
tcp\_end = 面向下游训练或操作语义的物理 end pose

需要注意，raw\_end 不一定可信，也不一定基于 base\_link 坐标系。因此在使用前需要先归一化，并与 verify\_end 做差异校验。



### **1\.5 正向运动学 FK**

FK，即 Forward Kinematics，表示根据机器人 joint 自由度值计算 end pose 的过程。这里的 end pose 通常指末端执行器或参考末端在指定坐标系下的位置和姿态。



对于一条从 base\_link 到末端 link 的关节链，可以表示为：

base\_link \-\> joint1 \-\> joint2 \-\> \.\.\. \-\> end



其矩阵形式为：

$T_{base \rightarrow end} = T_1 T_2 \cdots T_n$

- $T_i$ 表示第 $i$ 个 joint 带来的坐标变换。

- $T_1$ 就是parent\_link为base\_link的joint的origin变换矩阵 \* joint当前的运动变换，然后$T_n$就是最后连接到end link的joint的origin变换矩阵 \* joint当前的运动变换。

- 对 fixed 类型的 joint: $T_i = T_{origin}$

- 对 revolute / continuous 类型的 joint: $T_i = T_{origin} * R(axis, q_i)$，其中 $R(axis, q_i)$ 表示绕 axis 旋转 $q_i$ 单位为rad的旋转对应的变换矩阵。

- 对 prismatic 类型的 joint：$T_i = T_{origin_i} \cdot \mathrm{Trans}(axis_i \cdot d_i)$，其中 $\mathrm{Trans}(axis_i \cdot d_i)$ 表示沿着 axis\_i 平移 d\_i 单位为m的平移变换矩阵。



### **1\.6 Raw End 的坐标系归一化**

厂商 H5 中的 raw end pose 不一定直接基于 base\_link。例如某些构型的 raw end 可能是基于手腕、上半身、某个中间 link 或厂商自定义 frame 发布的。

因此，需要将 raw end pose 统一转换到 base\_link 坐标系下，这里的计算方法为：$ T_{base \rightarrow end} = T_{base \rightarrow raw} T_{raw \rightarrow end}$

其中：

- $T_{raw \rightarrow end}$：H5 中读取到的 raw end pose，也就是厂商rosbag话题中传的end pose, 可能会使用厂商自己的坐标系定义，比如右手的end pose使用了右臂的某个 link frame 作为坐标系

- $T_{base \rightarrow raw}$：raw\_pose被厂商定义的坐标系相对于 base\_link 的变换。

- $T_{base \rightarrow end}$：归一化后的 end pose，使用base\_link作为坐标系原点\.

举例来说，如果厂商提供的 joints/state/end 表示的是：

raw\_pose\_frame \-\> end



而 raw\_pose\_frame 本身不是 base\_link，则 normalize 阶段会先根据配置计算或读取：

base\_link \-\> raw\_pose\_frame



然后再计算：

base\_link \-\> end = base\_link \-\> raw\_pose\_frame \-\> end



最终将结果写入统一的 base\_link 坐标系下。



### **1\.7 TCP 偏置**

URDF 中的最后一个 link 不一定等于真实抓取点。例如夹爪的 gripper\_link 通常只是夹爪或灵巧手的安装座，而真实操作点应位于夹抓夹取时的接触点或灵巧手的指尖。为了在训练和操作语义中使用真实的 TCP，需要人工标定一个偏置，也就是从 URDF 的最后一个 link 到真实 TCP 的变换。



因此，本方案中使用 tcp\_from\_reference 描述从参考 link 到真实 TCP 的人工标定偏置：

$T_{base \rightarrow tcp}$ = $T_{base \rightarrow reference}$$T_{reference \rightarrow tcp}$

其中：

- $T_{base \rightarrow reference}$：由 state 关节数据和 URDF FK 计算得到。

- $T_{reference \rightarrow tcp}$：由配置文件人工标定。

- $T_{base \rightarrow tcp}$：最终的 tcp\_end。

有关具体标定的几种方案会在后续文档中讨论



### **1\.8 Raw End 与 Verify End 的误差校验**

为了判断厂商提供的 raw end 是否可信，方案中会通过计算raw end和verify end的误差来判断（主要用于自动化机审部分），以及在foxglove中可视化两者的差异，来辅助人工判断。

raw\_action\_end  vs verify\_action\_end
raw\_state\_end   vs verify\_state\_end



位置误差使用两者 position 的欧氏距离，单位为m：

$e_p = \lVert p_{raw} - p_{verify} \rVert$

位置误差混入了系统性偏置 直接 \|\|p\_raw \- p\_verify\|\| 会包含传感器安装偏移，要根据厂商p\_raw的测量方式做调整

`姿态误差使用两个四元数之间的夹角，单位为rad：`

$e_q = 2\arccos\left(\left|q_{raw} \cdot q_{verify}\right|\right)$



其中，使用绝对值是因为四元数 $q$ 和 $-q$ 表示同一个旋转。



这些误差首先会被用于判断厂商提供的 raw\_end 是否可用，然后也会在后续 collect 阶段写入质量报告，并在 inspect 阶段根据配置阈值判断这条数据是否通过。

### **1\.9 本方案中的配置文件角色**

当前方案主要涉及两个配置文件（这里的角色和功能后续可能会更改，需要及时更新）：

|配置文件|作用|
|---|---|
|robot\.yaml|描述 URDF、base\_link、关节 limit、H5 channel 到 URDF joint 的映射关系。|
|end\_config\.yaml|描述 raw end、verify end、tcp end 的计算方式、raw\_pose\_frame、输出路径和 TCP 偏置。|

这样的最终目标是可以确保hpc\_executor可以不直接依赖与urdf文件，而是完全通过 robot\.yaml 和 end\_config\.yaml 来完成 FK 计算和 end pose 的归一化。

### 2\.2\.  离线配置链路\(h5\_tf\_exporter\)

离线阶段的目标是为每个机器人构型生成一组经过人工确认的配置文件。这些配置文件后续会被生产流水线直接使用，因此需要在上线前完成 URDF、H5 joint mapping、end 定义和可视化结果的校验。

离线阶段的主要处理包括：

1. 使用 h5\_tf\_exporter 基于 URDF 和样例 H5 生成或更新配置文件模板。

2. 人工补全和修改 robot\.yaml 与 end\_config\.yaml。

3. 运行 h5\_tf\_exporter，生成 end\_summary\.md、report\.md 和 output\.mcap。

4. 使用 Foxglove 加载 MCAP 和 URDF，人工检查关节映射、夹爪方向、URDF 结构和 end pose 是否符合预期。

5. 如果发现配置问题，则回到配置修改步骤继续调整。

6. 确认无误后，将配置文件上传到 OBS，供生产流水线使用。

离线阶段的核心产物包括：

- robot\.yaml

- end\_config\.yaml

- end\_summary\.md

- report\.md

- output\.mcap

其中，robot\.yaml 主要描述 H5 joint channel 到 URDF joint 的映射关系、joint limit 和构型基础信息；end\_config\.yaml 主要描述 raw end、verify end、tcp end 的计算方式、raw\_pose\_frame 归一化方式和 TCP 偏置。



### 2\.3\. 生产处理链路

生产处理链路由 hpc\_executor 执行，主要包括 normalize、collect、inspect 和 align 四个阶段。

#### 2\.3\.1 normalize 阶段

normalize 阶段负责读取 OBS 中已确认的构型配置，并基于配置计算 base\_link 坐标系下的 end pose。

该阶段主要处理内容包括：

1. 根据构型 ID 从 OBS 下载对应的 robot\.yaml、end\_config\.yaml 和 URDF。

2. 读取 H5 中的 action/state joint 数据。

3. 读取 H5 中厂商提供的 raw end pose。

4. 对 raw\_action\_end 和 raw\_state\_end 做坐标系归一化，将其转换到 base\_link 坐标系下。

5. 基于 URDF、H5 joint 数据和配置文件，通过 FK 计算 verify\_action\_end、verify\_state\_end 和 tcp\_end。

6. 将计算结果写回 H5。

写回路径统一使用：

```YAML
joints/{action|state}/{joint_name}_end/{position|orientation}
    - attribute:
        - name: {end在urdf中对应的link}
        - end_type: [raw_end, verify_end, tcp_end]
```

需要注意，raw\_end 不是通过 FK 重新计算得到的，而是从厂商原始 end pose 读取后，根据配置进行坐标系归一化，所以部分没有提供end话题的构型这里会留空。verify\_end 和 tcp\_end 才是基于 URDF 和关节数据计算得到的结果。

#### 2\.3\.2 collect 阶段

collect 阶段负责统计 end 相关质量信息，并写入质量报告。

目前可以确定的质量指标包括：

- raw\_action\_end 与 verify\_action\_end 的位置误差和姿态误差。

- raw\_state\_end 与 verify\_state\_end 的位置误差和姿态误差。

- 使用 end 误差稳定性算法来分类误差的类型，分类为系统误差，非系统误差，低误差三种类型，并配上文本说明

- 各类 end 是否存在（检查metadata的字段是否存在）

- end pose 的基础统计信息。

- 配置或数据缺失导致的异常信息。

#### 2\.3\.3 inspect 阶段

inspect 阶段负责根据配置阈值校验 raw\_end 和 verify\_end 之间的差异，并将校验结果写入质量报告或质量标记。

该阶段主要处理内容包括：

1. 读取 collect 阶段生成的 end 质量统计结果。

2. 根据 inspect 配置中的阈值判断 position error 和 orientation error 是否在允许范围内。

3. 如果识别出可能有系统误差或者非系统误差过大，需要在error\_msg中汇报

4. 将校验结果写入质量报告或对应质量字段。

#### 2\.3\.4 align 阶段

align 阶段负责将前面阶段生成的 end pose 整理为可以用于交付的 H5 格式。

该阶段主要处理内容包括：

1. 读取 normalize 阶段写入的 end pose。

2. 合并action和state中不同joint\_name\_end为end group, 合并逻辑为优先使用合法的raw\_end, 没有的话就回退到使用verify\_end

3. 最终action和state只会各自保留一个命名为"end"的group, 用于最终交付

在本方案中，normalize 阶段会保留完整的中间 end 结果，用于内部的质量统计和校验；align 阶段则面向数据交付场景，只输出最终需要的 end pose 格式。

```YAML
joints/{action|state}/end/{position|orientation}
    - attribute:
        - name: [{end在urdf中对应的link_1}, {end在urdf中对应的link_2}]
        - end_type: [raw_end, verify_end, tcp_end]
```
